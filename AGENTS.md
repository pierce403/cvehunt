## Project Overview

CVEHunt is intended to become a fully autonomous CVE exploitability and remediation validation system. The long-term goal is one end-to-end job that can collect CVE context, build an isolated vulnerable/patched harness, prove exploitability with a PoC in that harness, prove the remediation blocks the same behavior, and emit durable evidence for defenders.

This direction is important: proving that a fix works requires more than metadata or a severity score. The pipeline should be able to demonstrate the vulnerable behavior, demonstrate the patched behavior, preserve the traces and artifacts, and produce an explainable judgement that downstream teams can audit.

## Honest Adversarial Loop Contract (v2 — in force)

A run must never claim a defensive signal it did not observe. The previous workflow credited itself for *artifacts existing* (a scaffolded PoC, a promoted fix, two fixture strings) regardless of whether anything actually executed. That is fixed. The contract a run must satisfy:

1. **Provision (health gate).** Before exploit development, the harness is built, started, and health-checked. The vulnerable surface must be *servable* (HTTP readiness + an instrumented probe that returns a known response shape) before the loop proceeds. A harness whose container just `console.log`s and exits is recorded `not_servable` and the run does NOT proceed to claiming an exploit. The provision result is logged to `provision/provision.json` and `provision/provision.log`.
2. **Exploit loop.** Against the running healthy vulnerable target, attempts are executed and observed. Each attempt is logged to `negotiation/exploit-rounds.ndjson` with `{round, attempt, request, response, observation, escalated}`. The loop iterates (bounded) until the CVE-described escalation is observed, not until a template file exists.
3. **Defend loop.** A candidate fix is applied, the patched target is rebuilt/restarted, health-checked, and the *same* exploit primitives are re-run against it. Each defense round is logged to `negotiation/defense-rounds.ndjson`. The patch is only counted as effective if the patched target *blocks* the escalation that succeeded against the vulnerable target.
4. **Adversarial residual rounds.** The exploiter attacks the patched target with a bounded set of residual/variant primitives (logged to `negotiation/residual-rounds.ndjson`). The defender iterates if anything re-escalates. The loop terminates when the exploiter cannot re-escalate within the bound, or a residual bypass is found.
5. **Verdict is behavioral.** A `negotiation/verdict.json` records `escalation_achieved`, `patch_effective`, `residual_bypass`, `rounds_total`. Only "escalation observed AND patch blocks it AND no residual bypass" yields `defensive_signal_observed` at ≥0.90–0.95. With no behavioral outcomes at all the verdict is `needs_human_review` (≤0.50) and is explicitly NOT a defensive signal — even if sources were acquired, a harness was scaffolded, and a fix was promoted.

This contract is enforced by regression tests in `tests/test_workflow.py` (search for `test_no_behavior_run_is_not_defensive_signal`, `test_provision_gate_refuses_non_serving_harness`, `test_adversarial_loop_records_rounds_and_verdict`, `test_residual_bypass_downgrades_verdict`).

### Known mislabeling that is now fixed

- `JudgeAgent` previously bumped confidence to 0.85/0.92 based on *artifact existence* (`poc_scaffolded`, `fix_generated`) before any behavioral outcome was consulted, so a run whose PoC never ran was labeled `defensive_signal_observed @ 0.92`. Confidence tiers now depend solely on `ExploitOutcome` / `NegotiationLog` behavior.
- `ValidatorAgent`'s `patched-vs-vulnerable differential check` previously passed whenever the two `cve.safe_fixture` strings differed — i.e., it credited input strings as evidence. It now passes only when a real behavioral differential was observed (vulnerable escalated AND patched blocked).
- `HarnessBuilderAgent._node_dockerfile` generated a `CMD node -e "console.log(...)"` stub that printed "ready" and exited; nothing served a port. React2Shell now has a source-backed instrumented Node target, and the provisioning gate requires both readiness and `/__cvehunt/probe` before a target is `servable`.
- The `exploiter/run-poc.sh` template previously `exit 2`-aborted on upstream readiness failure *before* ever running the PoC, so even when a shim was available it was never exercised. It now records per-target provision health and still runs the PoC against whatever is servable.

The intended pipeline shape is:

1. Collector: gathers CVE metadata, source context, version information, and available patch signals.
2. Researcher: derives vulnerability hypotheses, affected surfaces, and candidate validation primitives.
3. Harness Builder: creates isolated vulnerable and patched test environments for the target CVE.
4. PoC Developer: develops a proof-of-concept that is scoped to the isolated harness and used to validate exploitability.
5. Validator: runs the PoC against vulnerable and patched harnesses, captures evidence, and verifies that remediation changes the outcome.
6. Judge: reviews traces, artifacts, exploitability evidence, remediation evidence, and safety boundaries to produce an explainable assessment.

## Coding Conventions

- Python source lives under `src/cvehunt/`.
- Tests live under `tests/`.
- Prefer typed dataclasses for pipeline state and artifacts.
- Keep agent outputs structured. Use plain strings only for human-facing summaries.
- Preserve safety boundaries in tests when adding workflow behavior.
- CVE workdirs are stored under `cves/<CVE-ID>/` and are versioned so the public dashboard can link to metadata, traces, and reports in GitHub.
- Each persisted pipeline run should write `cve.json`, `trace.jsonl`, `pipeline_status.json`, `report.json`, and `report.md` under `cves/<CVE-ID>/runs/<RUN-ID>/`.
- Every persisted run gets a `run_score` in `pipeline_status.json` and `report.md`; 100 requires a vulnerable-target PoC trigger, patched-target block, candidate fix, and candidate fix validation. Lower scores reflect partial progress through source acquisition, harnessing, PoC generation/execution, patch generation, and validation. Reaching 100 is itself not enough to be `defensive_signal_observed` if a residual bypass was later observed — the Judge honors the `NegotiationLog.residual_bypass` flag.
- Root-level report artifacts under `cves/<CVE-ID>/` are reserved for fully successful end-to-end runs; incomplete runs stay archived under `runs/<RUN-ID>/`.
- GitHub Pages publishes from `docs/`; `docs/index.html` and `docs/CNAME` are generated by the React/Vite build.
- The public site is now a React/Vite app under `web/`; `npm run build` runs `scripts/generate_site_data.py`, writes `web/public/data/cves.json`, and builds into `docs/`. The site data includes both latest-CVE rows and an all-runs leaderboard sorted by run score.
- `FEATURES.md` follows the `features.md` convention: keep feature entries explicit about `stable`, `in-progress`, or `planned`, and attach testable properties plus checkable criteria.
- `CONTRIBUTING.md` is the contributor handoff doc for fork/test/run/PR workflow, including model-backed evaluation runs.
- `.github/workflows/pages.yml` runs uv tests plus the Vite build and deploys GitHub Pages on pushes to `main`.
- Local `gh` OAuth push rejected workflow-file commits without `workflow` scope; the GitHub app connector was able to create `.github/workflows/pages.yml`, then `git pull --ff-only` synced it locally.
- Use `uv run cvehunt run CVE-2025-55182 --persist --json --model <model-label>` when comparing models. Reports include `Run ID` and `Model`. SOURCE+harness-only runs (no `--execute-poc`, or execution that does not produce a behavioral differential) are now judged `needs_human_review` (≤0.50) and are NOT a defensive signal; `defensive_signal_observed` ≥0.90 requires a real escalation + a patched-side block observed in the harness.
- Use `./contribute.sh <CVE-ID>` for the contributor loop. It detects installed harness CLIs from `codex`, `gemini`, `claude`, `opencode`, and `pi`, validates model names when the harness exposes a local catalog, runs an early target-isolation preflight, records attribution as `<harness>:<model>`, syncs missing Python or Node project dependencies when prompted, runs a persisted JSON workflow with `--execute-poc` enabled by default, invokes supported selected model CLIs afterward and extracts safety-checked model-authored artifacts under `model_attempt/`, writes `model_attempt/`, `contribution_audit.{json,md}`, `contribution-interaction.log`, `contribute-output.log`, and `isolation-preflight.log` into the run directory, and rebuilds the Vite dashboard unless `CVEHUNT_SKIP_BUILD=1` is set. Each documented `CVEHUNT_*` environment override has an equivalent flag, including `--cve`, `--harness`, `--model`, `--skip-install`, `--skip-build`, `--skip-git`, `--dry-run`, `--execute-poc`, `--skip-execute-poc`, `--skip-model`, `--model-timeout`, `--base-port`, and `--isolation-backend`.
- Keep `contribute.sh` compatible with older Bash versions; avoid Bash 4-only helpers such as `mapfile` and `${var^^}` because some collaborator machines still ship Bash 3.x.
- Codex model attribution must use real local catalog slugs from `codex debug models`; `gpt-5.5-cyber` is not in the verified local Codex catalog, while `gpt-5.5` is.
- `./contribute.sh` now performs real external model invocation for supported harnesses (`pi`, `codex`, `gemini`, best-effort `claude`) after the deterministic Python workflow persists artifacts. The invocation writes `model_attempt/prompt.md`, `transcript.txt`, `stderr.txt`, `response.md`, `command.txt`, `metadata.json`, and `extracted.json`; it extracts only allowlisted model-authored files (`notes.md`, `refusal.md`, `fix.patch`, `poc.py`, `validation_plan.md`, `safety.md`) under `model_attempt/` after safety checks. Extracted `poc.py` must use loopback URLs and must not read hosts from env/args/input. Set `CVEHUNT_SKIP_MODEL=1` / `--skip-model` to disable it. Set `CVEHUNT_EXECUTE_POC=0` / `--skip-execute-poc` to avoid the default local Docker PoC execution in contributor runs. An empty or timed-out `model_attempt` (`metadata.json` `status` != `ok`/`completed`, zero-byte `response.md`) is a model-failure note recorded in `contribution_audit.json` as `model_attempt: failed/timeout`, NOT counted as a successful model-authored contribution.
- `ISOLATION.md` is the target-environment policy. Docker/Compose is only the current userland package harness backend; kernel, container escape, Kubernetes escape, browser, and runtime-boundary CVEs should use disposable VM or microVM backends (QEMU/KVM, Firecracker, Cloud Hypervisor, Kata) once implemented. `CVEHUNT_ISOLATION_BACKEND` supports `docker`, `external-vm`, `firecracker`, and `qemu` preflight paths; only Docker execution is implemented today.
- The current implemented pipeline is no longer fixture-only for supported npm and pypi CVEs: `ResearcherAgent` downloads vulnerable and patched package archives (npm tarballs via the npm registry; pypi sdists via `https://pypi.org/pypi/<pkg>/<version>/json`), extracts them into `sources/`, and writes `research/source_diff.patch`. PyPI version specs must be a single resolvable version (e.g. `litellm 1.81.16`), not a range.
- `HarnessBuilderAgent` writes `harness/Dockerfile.vulnerable`, `harness/Dockerfile.patched`, `harness/docker-compose.yml`, `harness/build-images.sh`, and `harness/README.md` into each persisted run. Compose port bindings are forced to `127.0.0.1` only.
- `ExploiterAgent` generates a structured investigation (`exploiter/investigation.md` and `.json`), localhost-scoped PoC (`exploiter/poc.py`), and orchestration runner (`exploiter/run-poc.sh`) keyed on `finding.vulnerability_class` (`sql injection`, `unsafe deserialization`, `unsafe interpolation`). Investigation artifacts must record hypotheses, target surfaces, controls, expected blockers, and next experiments. PoC scripts hardcode loopback targets derived from `--base-port` and pass through `SafetyPolicy.assert_localhost_scoped` before being written. Add new templates by extending `_select_poc_template` and `_class_specific_investigation` in `agents.py`.
- `FixDeveloperAgent` promotes the upstream vulnerable→patched diff as the candidate fix at `fix/candidate.patch` with rationale at `fix/rationale.md`, then validates it by applying the patch to `fix/applied/` (a copied vulnerable source tree) using CVEHunt's in-process unified-diff applier and comparing changed files to the upstream patched tree by SHA-256. Validation artifacts are `fix/validation.json`, `fix/validation.md`, and `fix/apply.log`. A separate candidate-fixed container image is not implemented yet.
- `HarnessRunnerAgent` runs only when `cvehunt run --execute-poc` is set. It calls `bash exploiter/run-poc.sh` from the run directory, which builds the compose stack or uses direct `docker build`/`docker run` fallback when Docker Compose is unavailable, polls per-target `/health/readiness` plus `/__cvehunt/probe`, records `provision/provision.json` per service, runs `exploiter/poc.py` against whatever was servable, and tees `exploiter/outcome.json`. Container logs are captured to `exploiter/logs/compose.log` on every run. The runner refuses to execute if `docker version` fails — the run still completes with `status="scaffolded"` and `next_step` pointing at the install requirement.
- The adversarial loop (`ProvisionAgent` + `AdversarialLoopAgent`) is what actually proves and disproposes the bug. `ProvisionAgent` health-checks each started container and writes `provision/provision.{json,log}` with per-target `servable`/`not_servable` status; fallback probes also require `/__cvehunt/probe`, so readiness alone is not enough. `AdversarialLoopAgent` runs bounded exploit/defense/residual rounds against the servable surface and writes `negotiation/exploit-rounds.ndjson`, `negotiation/defense-rounds.ndjson`, `negotiation/residual-rounds.ndjson`, `negotiation/negotiation.log`, and `negotiation/verdict.json` (escalation_achieved, patch_effective, residual_bypass, rounds_total). The verdict — not artifact existence — drives the Judge. When no target is servable, the loop records `target_not_servable`, produces no outcomes, and the Judge returns `target_not_servable`/`needs_human_review` ≤0.50; it never returns `defensive_signal_observed`.
- The litellm harness (`CVE-2026-42208`) is the canonical SQLi target. `HarnessBuilderAgent` writes `harness/config.yaml` (master_key=`sk-harness-master`, dummy `model_list`, `database_url: os.environ/DATABASE_URL`), `harness/db-init.sql` (creates `litellm_vuln` and `litellm_patched`), and `harness/instrumented/litellm_target.py`. The Dockerfiles install from the acquired source tree at `/workspace/package`, copy the instrumented wrapper, and the compose stack/direct-Docker fallback starts `python /workspace/instrumented/litellm_target.py` on port 4000. The wrapper starts child LiteLLM on loopback inside the container, proxies normal API routes, and exposes `/health/readiness` plus `/__cvehunt/probe`; the probe seeds a harness key through `/key/generate` so provisioning proves the proxy, DB, and auth setup are functional.
- When the runner produces `exploiter/outcome.json`, `ValidatorAgent` emits two extra evidence rows — `harness poc triggered vulnerable container` and `harness poc blocked by patched container` — and `JudgeAgent` upgrades confidence to 0.95 if both pass (or 0.88 if only one side does). PoC templates must emit `summarize()`-shaped records (`triggered: bool`, `detail: str`) for the runner to parse them. Manual runs on `CVE-2026-42208` with `--base-port 4100` built images, started containers, seeded local LiteLLM virtual keys, and produced real outcomes; the upstream LiteLLM containers both returned 401, while the shim vulnerable/patched pair demonstrated the SQLi class differential. Do not mark this as 100/100 until a real upstream vulnerable-container trigger is observed.
- For supported vulnerability classes (currently `sql injection`), `HarnessBuilderAgent` also emits a "shim" mini-harness under `harness/shim/{vulnerable,patched}/`: a tiny FastAPI service paired with a sanitized counterpart that demonstrates the class deterministically. The shim seeds a single synthetic credential into an in-container sqlite DB and exposes `/verify`. Compose adds `shim-vulnerable` (127.0.0.1:4010) and `shim-patched` (127.0.0.1:4011); the SQLi PoC probes both. Two extra evidence rows feed Judge — `harness shim triggered vulnerable demo surface` and `harness shim blocked by patched demo surface`. A shim-only differential (vulnerable shim escalates while patched shim blocks) reaches `defensive_signal_observed` capped at 0.90 and the rationale must state it is a class-level demonstration, not an upstream-package exploit proof. The shim is only exercised when `--execute-poc` runs and the shim containers pass provisioning.
- Judge's "all evidence must pass" gate now applies only to *structural* evidence (artifacts materialized, harness built, candidate fix promoted). The four outcome-derived rows above are *behavioral*: they inform confidence and rationale but can legitimately fail without flipping the run to `insufficient_evidence`. New outcome-based check names must be added to `behavioral_check_names` in `JudgeAgent.judge` to keep that distinction working.
- `SafetyPolicy` (and the contribute.sh model-attempt extractor) enforce ONLY the operational targeting boundary: PoC artifacts target the loopback harness (`127.0.0.1`/`localhost`/`::1`) only, never real third-party infrastructure, and `poc.py` must hardcode loopback hosts and must not read them from args/env/input. CVEHunt's whole point is to fully explore attacker capabilities, so the pipeline deliberately does NOT content-filter attacker-capability vocabulary. Reverse shells, credential exfiltration, persistence, privilege escalation, and code execution are all fair game to describe and to demonstrate against the loopback harness if the CVE's impact makes them reachable. An earlier version substring-blocked `reverse shell`/`bind shell`/`weaponize` in model prose and short-circuited artifact extraction on any hit — which destroyed safe outputs the audit needed (e.g. it nuked GLM 5.2's `notes.md`/`validation_plan.md`/`safety.md` because GLM's own `safety.md` declared "No reverse shell, bind shell, ..." and the scanner matched the phrase *inside the negation*). That prose scanner is removed entirely; the only remaining enforcement is per-artifact loopback/host-source checks (operational), not vocabulary.

With `--execute-poc`, `HarnessBuilderAgent` emits `harness/instrumented/react2shell-server.js`; the Node target exposes `/health/readiness`, `/__cvehunt/probe`, and `/server-function`. Docker Compose is not installed (`docker compose version` failed), so `run-poc.sh` now falls back to direct `docker build`/`docker run`. `uv run cvehunt run` only builds/runs images when `--execute-poc` is set; `./contribute.sh` sets that by default unless `--skip-execute-poc` or `CVEHUNT_EXECUTE_POC=0` is used.
- Local port 4001 is occupied by an `ipfs` process in this environment; use `--base-port 4100` or another free loopback pair for executed harness runs.
- Model-attempt runs (verified 2026-06-18): `pi` is invoked with `--mode json --thinking minimal` (default, override `CVEHUNT_PI_THINKING`). ## Known Issues & Solutions

- `python` may not be installed in this environment; use `python3` or `uv run python`.
- `uv run pytest` failed here because the pytest console script could not be spawned; `uv run python -m pytest` passed.
- Public site notes from the initial MOAK research are in `moak_site_notes/` and are intentionally not part of the package.
- Regenerate the public site with `npm run build` after syncing or running CVEs so `docs/data/cves.json` reflects the current root-level `cves/` workdirs.
- The React site now reads phase-level status from `pipeline_status.json`; if a stage is stubbed or not implemented, fix the run data rather than trying to infer progress from `trace.jsonl` length alone.
- The dashboard now uses an inline accordion on the main CVE table: clicking a CVE expands the latest run summary in place, while `#/cve/<CVE-ID>` remains the deep-link detail route.
- For the inline dashboard accordion, the whole summary row should toggle open/closed; artifact links inside the row should stop propagation so they remain direct navigation targets.
- NVD may lag behind newly published advisories. If a CVE like `CVE-2026-42208` has no NVD record yet, it is acceptable to add `cves/<CVE-ID>/cve.json` from a primary advisory source such as a GitHub advisory, then rebuild the site so it appears as tracked but not analyzed.

## Agent Tips

- Read `README.md` for the product framing before changing behavior.
- If adding real integrations later, keep them behind explicit adapters and test with fixtures first.
- PoC artifacts must stay harness-bound: hardcoded `127.0.0.1` targets, no environment-overridable hosts, no credential exfiltration. `SafetyPolicy.assert_localhost_scoped` is the runtime check; keep templates routed through it.

- Pi project-local extensions auto-discover from `.pi/extensions/`; async `pi.registerProvider()` providers appear in `pi --list-models` without extra settings.
